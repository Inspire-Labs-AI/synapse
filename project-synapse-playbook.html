01 · System Prompt

The Charter: What Synapse Is

Synapse is an AI codebase-intelligence, visual-topology, and product-blueprinting engine. Not a linter. Not a docs generator you run once and forget. Not a chatbot bolted onto a repo. It exists to turn a codebase into a substrate you can reason over, and that value lands in two equally serious places.

The mandate is simple to say and hard to earn: make any codebase legible, answerable, and buildable-against. Some of that is comprehension, helping a person hold an unfamiliar system in their head. Some of it is construction, helping them plan the next feature against what already exists. Neither is the "real" work and neither is the leftover. They're two directions of one job.

Mandate 1 · Understand

Take a repository nobody on the team has time to read and make it navigable: a visual dependency map, a chat that traces where a route is handled, a guided tour that walks a new engineer through the code dependency-first, and dead-code detection that says which files are safe to question. Comprehension you'd otherwise buy with weeks of ramp-up.

Mandate 2 · Build

Take a feature idea and score it against the real codebase: what already exists to reuse, what to extend, and what to build new, and where it belongs. Synapse answers "should we build this?" and "here's the blueprint" from the actual structure of your code, not a generic template.

What winning looks like

An engineer opens a repo they've never seen and, within an afternoon, can name its entry points, trace a request, find its dead weight, and draft a grounded plan for the next feature. A map that dazzles in a screenshot but can't answer a real question isn't a milestone, it's a screensaver. Synapse is judged on answers, not aesthetics.

★ The bar is grounding

The frontier here isn't a prettier graph, it's trust. Anyone can render 800 nodes. The hard part is that every claim, every highlighted file, every "reuse this", is backed by the real AST and the real source. Synapse is built to be believed, which is why it verifies before it asserts.

02 · System Prompt

The Core Loop: Ingest → Graph → Understand → Build

Everything Synapse does is one loop. Source goes in; a fused structural-and-semantic graph comes out; and every surface, the map, the chat, the blueprint, the prune report, is a different question asked of that same graph. Get this loop in your head and the whole system stops being a pile of features and becomes one argument.

Ingest

A Go worker pool walks a local directory or a shallow-cloned repo, dispatches each file to its real parser by extension, and persists a uniform FileAnalysis: imports, exports, declarations, endpoints. Fast, concurrent, language-native.

Resolve & graph

Imports are resolved language-aware, TS relative paths and baseUrl, Go package→directory, Rust modules, into a true file→file dependency graph stored as ast_relationships. This is the skeleton.

Enrich

Structural chunks (functions, classes, methods) are embedded with a code-aware model into vector_chunks with an HNSW index. Now the graph carries meaning as well as shape, the semantic layer.

Understand

The engineer explores the neural map, asks the assistant a question (hybrid vector + keyword RAG that traces the answer onto the canvas), takes a dependency-ordered tour, or scans for dead code, all reads of the same graph.

Build

A feature idea is decomposed into intents, each scored green/yellow/red against the codebase, and returned as a validation ("should we build this?") or a roadmap ("what to reuse, extend & create, and where"). The loop closes: understanding becomes a plan.

03 · The Neural Model

The Neural Model: Five Cybernetic Capabilities

The neuroscience naming isn't a coat of paint. Each term names a real capability with real behaviour. Learn the five and you know the platform's shape, because every feature ladders up to one of them.

Cortex Perspectives View layer

Persona-adaptive zoom. Synaptic is the default deep dive: individual file cards, function anchors, raw structural dependency lines, the engineer's view. Executive collapses files into parent folder blocks and macro-components with business-flow edges, for a PM or a lead who wants the shape, not the wiring. One graph, two altitudes.

Axon Pathways Onboarding

A guided, dependency-ordered reading tour. Synapse computes a topological sort of the graph, curates the most instructive files (foundations, entry points, high-fan-in modules), and flies the camera through them step by step, foundations first, with a role, a summary, and the key symbols for each. New-engineer ramp-up as a cinematic walkthrough, playable hands-free.

Dendrite Callouts Pattern layer

Contextual pattern identification. As the parser reads code, it can flag complex idioms, closures, higher-order functions, custom decorators, transactional blocks, and decorate the canvas with descriptive markers so a reader sees not just what connects to what, but how the code is written.

Myelin Insulation Docs layer

Human documentation, woven in. The ingestion walker scales to markdown, internal wikis, and /docs. It vectorizes those segments as myelin_doc chunks and overlays human-written context directly onto the automated code graph, so the prose that explains the code lives next to the code.

Synaptic Engrams Portability

Offline memory. A serialization layer exports the active graph into a compressed, Git-portable JSON snapshot, so a teammate can load an identical canvas without a live database. The map becomes a file you can commit.

◆ The through-line

Cortex changes the altitude, Axon changes the path, Dendrite adds the idiom, Myelin adds the prose, and Engram makes it portable. Five lenses, one graph. When a new capability is proposed, the first question is which of these five it extends.

04 · The Substrate

The Substrate: Architecture & Data Model

Every surface is a view; this is the thing they're views of. A local monorepo monolith: a Go engine for concurrent parsing and retrieval, a Next.js 16 canvas for the map and the panels, and PostgreSQL + pgvector holding both the structure and the meaning.

Backend · Go

Ultra-fast, concurrent AST processing and ingestion. A worker pool parses, resolves, chunks, and embeds; native pgx pools talk to Postgres; the RAG, blueprint, prune, docs, architecture, and axon engines all read the same store.

Frontend · Next.js 16

A TypeScript App-Router workspace. The graph is a d3 force + radial layout rendered on canvas (not SVG, so 800+ nodes stay smooth), with GSAP for camera motion and streamed panels for chat and blueprint.

Store · PostgreSQL + pgvector

Three tables carry everything: code_files (the nodes), ast_relationships (the edges), and vector_chunks (the meaning), the last with an HNSW index for fast semantic search. Runs locally via Docker.

Intelligence · pluggable

Embeddings auto-chain across providers (Jina code vectors, OpenAI, Gemini, local Ollama, deterministic floor); the LLM layer routes per engine. Swap a key in .env and the provider changes with no other edit.

The one data model to hold in your head

If you remember nothing else about the internals, remember these three tables. Every feature is a query over them:

TableIs the graph's…Holds
code_filesnodesevery ingested file: path, language, root. One row = one node on the canvas.
ast_relationshipsedgesimports, exports, endpoint, and intra-file calls, the resolved, directed dependency graph.
vector_chunksmeaningfunction/class-level chunks + their embeddings (HNSW), plus myelin_doc chunks for markdown.

The layout

project-synapse/ ├─ backend/ # Go: engines, workers, parsers, RAG │ ├─ internal/parser/ # MultiParser: tsc · go/parser · rust lexer · resolve │ ├─ internal/ingest/ # worker pool · enrich · embeddings │ ├─ internal/store/ # pgx: files · relationships · vector_chunks │ ├─ internal/rag/ # hybrid retrieval + streamed answers │ ├─ internal/blueprint/ # validate / roadmap engine │ ├─ internal/prune/ # dead-code + LLM verification │ └─ internal/{docs,architecture,axon,bugs}/ ├─ frontend/ # Next.js 16 workspace + d3 canvas │ └─ app/workspace/ # GraphView · Assistant · Blueprint · Prune ├─ docker/ # Postgres + pgvector, schema init └─ docs/ # architecture logs · API registry

And the shape of the contract every parser produces, one structural payload every downstream stage consumes uniformly:

FileAnalysis · the universal contract
type FileAnalysis struct {
  RelPath      string
  Language     string       // typescript · go · rust · markdown
  Imports      []ImportRef  // resolved: relative · baseUrl · alias · pkg
  Exports      []Symbol     // the file's public surface
  Declarations []Decl       // functions, classes, methods → chunks
  Endpoints    []Route      // HTTP routes → graph entry points
}
// one contract. every stage — resolve, chunk, graph, RAG,
// blueprint, prune, axon — reads it the same way.
05 · The Surfaces

The Surfaces: What You Actually Touch

Six views onto the one graph. Each answers a different question, but none of them invents; they all read the substrate from §04. Here's what each is for and when to reach for it.

The Neural Map Galaxy · Radial

The centrepiece. A Galaxy force-directed cloud clusters files by module and colours them by folder; a Radial hierarchy arranges them on a rim with bundled dependency arcs. Hover a node for its imports/importers at a glance; click to pin it and watch its dependencies stream, amber outward (what it imports), cyan inward (what imports it). Search to fly to any file; isolate a module from the legend; double-click to frame a neighbourhood. Rendered on canvas so it stays fluid at scale.

The Assistant Hybrid RAG

Ask the codebase a question in plain language, "where is the /category route handled?", and Synapse runs hybrid retrieval (vector similarity + keyword over files and relationships), streams a grounded answer, and traces it onto the map: the relevant files light up, the execution flow fills the dock. Click a node to open its detail drawer, a short AI summary of the file's role plus its imports and importers.

The Blueprint Validate · Roadmap

Pitch a feature; pick a mode. Validate returns a decision, a verdict (Build / Build later / Reconsider), the reuse leverage, the product impact, and the effort and risks. Roadmap returns the plan, what to reuse, which files to extend, and the new files and folders to create, following your repo's real conventions. Both are grounded in a green/yellow/red scoring of each intent against the actual code.

Prune Dead code

Synaptic pruning: a multi-signal dead-code scan over the AST graph, orphan files, dead clusters, unused exports, then an LLM verification pass that reads the real source to separate framework-invoked files from genuine husks. Findings come with a confidence and a concrete, source-grounded reason, and are always candidates for review, never automatic deletions.

Docs & Architecture Synthesis

Generated, grounded documentation and an interactive architecture diagram, both written from the real graph and its retrieved code, so the docs describe the system as it actually is (the right state manager, the real data flow), not as a generic template imagines it.

▲ How to pick a surface

New to the repo? Start on the map and take an Axon tour. Have a specific question? Ask the assistant. Cleaning up? Run prune. Planning the next feature? Open the blueprint. They're not separate apps, they're four questions asked of the same graph.

06 · Runbooks

Playbooks: Driving Synapse

The concrete moves. Each runbook names the job, the surface, the steps, and what "done" looks like. Don't read these as reference, run them, they're the fastest way to internalize how the engine behaves.

★ Before any of these

Bring the stack up first: Docker (Postgres + pgvector) must be running, then the Go backend on :8080, then the Next.js workspace. Ingestion is at parse-time, so any change to how code is read only takes effect on the next ingest.

Playbook 01 · Onboarding a repo

From URL to Living Map

Turn a repository you've never seen into a navigable neural graph.
Job
Ingest a local directory or a shallow-cloned repo and get a canvas you can explore.
Steps
  1. Point Synapse at a local path (or clone-on-demand with a personal access token).
  2. The worker pool walks the tree, parses each file with its native parser, and resolves imports.
  3. The enrich pass chunks declarations and embeds them into the semantic layer.
  4. Open the workspace; the graph settles into a Galaxy of module-coloured clusters.
  5. Switch the repo from the selector; the map, assistant, and blueprint all scope to it.
Done when
The node/edge count matches the repo, entry points read as reachable, and the assistant can answer a question about it.
Playbook 02 · Comprehension

Read a Codebase in an Afternoon

Hold an unfamiliar system in your head without reading every file.
Job
Understand a repo's shape, its entry points, and how a request flows.
Steps
  1. Take an Axon tour, foundations first, camera flying step by step (hit play for hands-free).
  2. Switch to Executive perspective for the macro shape; back to Synaptic for the wiring.
  3. Ask the assistant where a specific route or symbol lives; watch it trace onto the map.
  4. Click any file for its AI summary, imports, and importers; follow the connection chips to hop the dependency chain.
Done when
You can name the entry points, trace one request end to end, and point to the core modules.
Playbook 03 · Planning

Validate, Then Blueprint a Feature

Decide whether to build something, then get the plan, grounded in your code.
Job
Turn a feature idea into a build/no-build call and an implementation plan.
Steps
  1. Open Blueprint, type the feature idea, and choose Validate.
  2. Read the verdict, reuse leverage, product impact, and effort/risks; decide.
  3. If it's a go, flip to Roadmap and re-run for the same idea.
  4. Follow the plan: reuse the named files, extend the listed ones, create the new files where it says, matching your repo's folder conventions.
Done when
You have a defensible verdict and a file-level plan that names real paths, reuse before build.
Playbook 04 · Cleanup

Hunt Dead Code, Safely

Find what's genuinely removable, with a reason you can trust.
Job
Surface orphan files, unused exports, and commented-out husks, without false positives.
Steps
  1. Open Prune; the structural scan runs over the AST graph (reachability from entry points).
  2. The LLM verification pass reads each candidate's real source and separates framework-invoked files from genuine dead code.
  3. Read each finding's confidence and its source-grounded reason ("fully commented out", "exports nothing referenced").
  4. Treat every finding as a review candidate, Synapse flags; a human decides.
Done when
You have a short, high-confidence list of removable files, each with a concrete justification.
▲ The pattern across all four

Each runbook is the same move in a different key: point Synapse at real code, let it read the true structure, and hand you a grounded answer, a map, a trace, a plan, a verdict, instead of a guess. That's the whole engine in miniature.

07 · Under the Hood

Under the Hood: The Engines

You don't need this to use Synapse, but knowing how it earns its answers is how you trust them, and how you extend it. Four engines do the real work.

EngineWhat it doesWhy it's trustworthy
MultiParser + resolverDispatches each file to a real parser, TS/JS via a tsc subprocess, Go via go/parser, Rust via a brace-aware scanner, then resolves imports language-aware (relative, baseUrl, tsconfig aliases, Go packages, Rust modules).True ASTs, not regex. The baseUrl resolution is why src/… imports actually connect instead of looking external.
Embedding chainChunks declarations to a structural boundary and embeds them; auto-selects a provider (code-aware Jina → OpenAI → Gemini → local Ollama → deterministic floor) and auto-migrates the vector column when the dimension changes.Code-specialized vectors, and a deterministic fallback means it always runs, even offline.
Hybrid RAGBlends vector similarity with keyword search over files and relationships, assembles a grounded context window, and streams the answer while emitting the highlighted files and execution flow.Retrieval is over real chunks; the answer is traced onto the graph, so you can check it against the map.

◆ The extension rule

Adding a language, a provider, or a surface? Route it through the existing contracts, the FileAnalysis payload, the three-table store, the embedding chain, before inventing a new path. Stay within the four supported languages and the raw-SQL-over-pgvector approach unless there's a real reason not to. The system's power is that everything reads one graph; a bypass weakens that.

08 · Operating Laws

The Weights: What Synapse Optimizes For

These are the principles behind every design call in the system, the weights that break ties. When a feature is ambiguous, or you're extending Synapse and unsure which way to go, re-read these. They decide it.

01

Structure before semantics

Parse the true AST first; infer meaning second. The dependency graph is ground truth, and the semantic layer is a lens over it, never a replacement for it. A beautiful embedding over a wrong graph is confidently wrong.

→ Test: is this claim backed by a real edge, or a vibe?

02

Ground every claim

The assistant retrieves real chunks and traces its answer onto the canvas. It never names a file that isn't there or a route that doesn't exist. If it can't ground a claim, it says so rather than inventing one.

→ Test: can the user click the answer and land on real code?

03

Reuse before build

The blueprint scores what already exists, green to reuse, yellow to extend, before it ever proposes a red "build new." The most valuable output is often "you already have this," and new files are the last resort, not the first.

→ Test: did we check what exists before suggesting new code?

04

Verify before you assert

Prune runs a structural signal, then reads the real source to confirm it. Dead code is flagged with a source-grounded reason and a confidence, never asserted from a filename. A false positive costs trust, and trust is the product.

→ Test: did we read the code, or just its name?

05

Human disposes

Synapse proposes; the engineer decides. Nothing destructive runs on its own, prune flags candidates, the blueprint suggests a plan, the assistant recommends. The last, irreversible call is always a person's.

→ Test: is the destructive action gated behind a human?

06

Language-native, always

Each language is read by its own real parser. No regex pretending to understand syntax it can't. When a new language arrives, it comes with a true parser and a real resolver, or it doesn't come at all.

→ Test: would this survive a weird-but-valid file?

07

One graph, many views

Every surface reads the same substrate, so they agree. A new capability is a new question asked of the graph, not a new source of truth. The moment two features disagree, one of them stopped reading the graph.

→ Test: does this read the graph, or a copy of it?

08

Legible over clever

The win is comprehension, so the interface earns its keep by making things clear: constant-size labels that don't overlap, focus that dims the noise, reasons written in plain language. A cleverer visualization that's harder to read is a regression.

→ Test: did this make the codebase easier to understand?
